Maverick Coding Platform - Agent Context Memory
Last updated: 2026-03-26

1) Project Overview
- Monorepo with FastAPI backend + React/Vite frontend + PostgreSQL + Redis.
- Goal: AI-assisted coding platform with resume-based onboarding, adaptive assessments,
  learning-path generation, leaderboard, hackathon workflows, and admin stats.
- Repo info:
  - Name: Maverick-Coding-Platform
  - Owner: Gowshik-S
  - Current branch: master
  - Default branch: main

2) High-Level Architecture
- Frontend (React + Vite): User onboarding, login, dashboard, leaderboard UX.
- Backend (FastAPI): Auth/register, assessment, recommendation, leaderboard,
  hackathon, admin APIs.
- PostgreSQL: System of record for users, assessments, learning paths, hackathons,
  submissions, badges.
- Redis: Fast state for progress events, current question cache, leaderboard sorted set.
- Groq LLM: Optional enrichment for skill extraction/question generation/grading/path;
  deterministic fallback logic is implemented when GROQ_API_KEY is absent.
- OCR stack: pytesseract + poppler + pillow + pymupdf + pdf2image for scanned resumes.

8) Auth and Onboarding Flow (important behavior)
- Frontend route flow:
  - /register -> /onboarding/resume -> /onboarding/manual (optional) -> /onboarding/success
- Register API behavior:
  - checks existing email in users table
  - if exists: HTTP 409 detail="Email already registered"
  - parses resume text (txt/pdf/docx/image, OCR if needed)
  - if incomplete profile and no extra fields:
    - returns needs_more_info with missing fields/questions
  - if complete or extra fields supplied:
    - inserts user, runs profile skill extraction, returns user_id and skills
- Login API behavior:
  - finds user by email
  - if missing: HTTP 404 detail="User not found"
  - if found: returns user_id/name/skills
  - note: password verification is currently stubbed (not enforced yet)

12) Known Gaps / Technical Debt
- Login password is placeholder (no hash/verify enforcement yet).
- Some endpoints still return {"error": ...} with 200 rather than HTTPException
  (for example parts of assessment/hackathon), so error-contract consistency can
  be further standardized.
- Redis/DB failures are partially tolerated for startup health route simplicity.
- LLM outputs rely on JSON extraction; fallbacks exist but should be monitored.

14) Important Conventions
- Fallback deterministic behavior is explicitly marked in code comments as
  hardcoded/stub.
- Backend uses psycopg2 direct SQL helpers and Redis wrapper helpers (no ORM).
- Frontend assumes JSON API responses; parseJsonSafe used to avoid hard crashes.
- Route prefix standard: /api for HTTP and websocket endpoints (except / and /health).
